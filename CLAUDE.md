## What this app is

Laravel 13 porta-potty-rental **lead generation platform**. The entire app is built to:
1. Rank city/state landing pages on Google (programmatic SEO).
2. Route the resulting phone calls through SignalWire IVR to paying buyers.
3. Let admins manage cities, buyers, phone numbers, call logs, invoices, and AI-generated content.

Domain-specific comments are sometimes written in Bengali — that is intentional, follow the existing pattern rather than translating or removing them.

## Common commands

```bash
composer run dev     # concurrent: artisan serve + queue:listen + pail + vite
composer run test    # config:clear + php artisan test
composer run setup   # full bootstrap (install, key:generate, migrate, build)

./vendor/bin/pint                         # auto-fix PHP style (run before commits)
./vendor/bin/pint --test                  # CI-style check

php artisan test --filter=test_name       # run a single test by method or class
./vendor/bin/phpunit tests/Feature/       # feature tests only
```

Tests run against `:memory:` SQLite (see `phpunit.xml`); dev DB is a SQLite file at `database/database.sqlite`.

## Multi-domain architecture (critical)

This is **not a single-tenant site**. It hosts multiple rental brands from the same codebase:

- `Domain` model holds each brand (slug_prefix, service_types, colors, business name, etc.).
- `DomainMiddleware` (registered in `bootstrap/app.php`) resolves the current domain from session → host → fallback first domain, and shares `$currentDomain` with every view.
- `Domain::current()` is the ambient "which brand are we serving right now?" lookup, used throughout helpers, services, and views.
- `App\Traits\BelongsToDomain` adds a **global scope** so queries only return rows tied to the current domain via the `domain_cities` pivot. Use `->withoutGlobalScope('domain')` (or the `allDomains()` / `withoutDomainScope()` scopes on the trait) when you genuinely need cross-domain data — e.g., in `Admin\Global*Controller` master-data screens.
- Admin has two parallel namespaces: `/admin/global/states` and `/admin/global/cities` manage the master catalog across all domains; the non-global `/admin/cities`, `/admin/states`, etc. operate scoped to the currently selected domain.

When adding a new model that should be domain-scoped, use the `BelongsToDomain` trait and ensure a row exists in `domain_cities` — otherwise records will silently disappear from all queries.

## Routing gotcha

`routes/web.php` ends with a catch-all:

```php
Route::get('/{slug}', [PageController::class, 'cityPage'])->where('slug', '[a-z0-9][a-z0-9\-]*');
```

This matches **any** single-segment path and dispatches to the service/city page handler. Any new public top-level route (e.g. `/pricing`, `/about`, `/blog`) must be declared **before** this line. The state page route is constrained to a known list of state slugs and also sits above the catch-all. Admin-logs routes are likewise placed before the catch-all intentionally.

SignalWire webhook routes under `/webhook/signalwire/*` are CSRF-exempt via `bootstrap/app.php` (`webhook/*` pattern).

## Content generation pipeline

Programmatic SEO content (city pages, FAQs, testimonials, blog posts) is generated by AI, not authored.

- `App\Services\MultiAiService` is the entry point. It iterates active `AiApiKey` rows by priority, skipping keys over their per-minute or per-day rate limits, and falls through provider-specific services (`AnthropicService`, `OpenAIService`, `GeminiService`, `GroqService`). Rate-limit and daily-limit errors flag the key automatically.
- `ContentGeneratorService` composes the prompts and writes the resulting `ServicePage`, `Faq`, `Testimonial`, `BlogPost` records.
- Long generations run as queued jobs: `GenerateCityContentJob`, `GenerateStateContentJob`. Admin controllers fire the job and expose a `*/generation-progress` endpoint that the UI polls.
- Scheduled automations live in `routes/console.php`: daily blog post (`blog:generate-daily`), daily batch of city pages (`city:generate-daily-pages`), plus Google Indexing API submit/check commands. The batch size for city generation is controlled by `DAILY_CITY_PAGE_GENERATION` env.

When adding a new generation flow, prefer extending `MultiAiService` + a dedicated service method over calling providers directly, so the rate-limit/failover logic is reused.

## Call tracking pipeline

- `PhoneNumber` rows belong to a `City`. Inbound calls hit `SignalWireWebhookController@incoming`, which returns SignalWire XML (LaML).
- IVR flow: `incoming` → `gather` (caller key-presses) → `whisper` (played to the buyer before connect) → `status` (final call record). Every step writes/updates a `CallLog`.
- `SignalWireService` encapsulates API calls and XML generation. Keep webhook controllers thin; put logic in the service.

## Caching

Homepage and page data are cached aggressively — keys and TTLs are listed in `AGENTS.md` §3. When editing cities, service pages, FAQs, or testimonials, invalidate or bust the relevant cache keys, or expect stale content in dev.

## Things to know before editing

- `composer run test` runs `artisan config:clear` first — necessary because cached config can poison tests. If tests pass locally but fail after config changes, try that.
- `app/helpers.php` is autoloaded via composer.json `files` and defines `phone_raw()`, `phone_display()`, `domain_cta_phone()`, etc. Prefer these over hardcoding phone numbers.
- There is a `ContentGeneratorService.php.backup` file in `app/Services/` — it is stale; ignore it.
- Authentication uses Laravel Breeze; admin area is `auth + verified` middleware.
- Frontend is Blade + Alpine.js + Tailwind v4 via Vite. No SPA framework.
